03Architecture and Technical Decisions

The tradeoffs I made and why I made them.

Scoring Engine: Pure JS, Zero API Calls

All 12 analysis features run in a single analyze() function that takes raw video data and returns a complete AnalysisResult. No network calls, no external services. This was deliberate. It means the scoring logic works offline, is testable in isolation, has zero additional latency, and costs nothing to run at scale. The YouTube API is used exactly twice: once for channel metadata, once for video statistics. Everything else is computed locally.

Log-Scale Outlier Scoring

The first version used a simple ratio: views / median. This broke badly for channels with mixed content. A channel with 30 Shorts averaging 5K views and 20 long-form videos averaging 2M views would have a median around 5K, making every long-form video score 10/10. Useless.

The fix: separate baselines for Shorts and Long-form, plus a log-scale transformation (5 + log2(ratio) * 1.5) that compresses the range. Now a video at the median scores exactly 5.0, double the median is 6.5, and you need 8x the median to hit 9.5. The scale has meaning. I spent real time getting this right because a wrong outlier score undermines trust in the entire product.

YouTube API Quota Efficiency

The YouTube Data API has a daily quota of 10,000 units. A naive implementation using search.list costs 100 units per call. I use playlistItems.list (the channel's uploads playlist) which costs just 1 unit, followed by a videos.list batch call for another 1 unit. Total cost per analysis: about 3 quota units instead of 100+. That means ~3,000 analyses per day on the free tier instead of ~100.

URL Parsing

YouTube URLs come in at least 8 formats: @handle, /channel/UCxxx, /c/customname, /user/username, youtu.be/videoId, ?v=videoId, /shorts/videoId, /live/videoId. I handle all of them. If someone pastes a video link, I resolve it to the channel first, then analyze the channel. This is the kind of detail that separates "it works in a demo" from "it works when a real user tries it."

Server Components for Data Fetching

The analysis page is a server component that fetches data before sending HTML to the client. No loading spinners, no client-side API calls, no CORS headaches. The YouTube API key stays server-side, never exposed to the browser. Client components handle interactivity only (sorting, filtering, modals).

What I Chose NOT to Build

  • Authentication: Login page exists as a shell but does not enforce auth. For a Monday demo, the founder needs to paste a URL and see results. Auth adds friction without adding value to the demo.
  • Database: No persistence layer. Saved analyses use localStorage. This is fine for an MVP. Adding a database before validating the product would be premature engineering.
  • Real-time updates: Data is fetched once per analysis. Polling or WebSocket updates would add complexity without changing the demo experience.

04Product Thinking

What I would add, what feels missing, and where this goes next.

What I Would Add With More Time

Multi-Channel Comparison. The biggest gap. Right now you analyze one channel at a time. The real power is putting two or three competitors side-by-side: who is winning on which days, whose formats are declining, where the content gaps are. I would add a split-view comparison mode where you paste 2-3 URLs and get a combined intelligence brief highlighting relative strengths and weaknesses.

Saved Analysis History with Trends. Right now, each analysis is a snapshot. If you analyzed the same channel last month, there is no way to compare. I would store historical results in a database and show trend lines: "Outlier score improved from 4.2 to 6.8 over 90 days" or "Posting cadence dropped from every 3 days to every 2 weeks." That is the kind of insight that justifies a $499/month subscription.

Topic Clustering. Group a channel's videos by topic using keyword extraction from titles and descriptions, not just format pattern matching. This would make the Format Graveyard smarter and enable a "Content Map" visualization showing which topics drive the most views.

Email Reports. The weekly scheduled report feature has a UI shell but no backend. With a simple cron job and email service (Resend or SendGrid), this would let agencies auto-deliver branded reports to their clients every Monday morning without logging in.

Comment Sentiment Analysis. Right now Ghost Audience just checks if comments exist. With even basic NLP, I could flag whether comments are positive ("amazing video"), negative ("clickbait"), or questions ("can you do a video about X?"). Questions in comments are a gold mine for content ideas.

What Feels Missing Right Now

  • Onboarding flow. A first-time user lands on the analysis page and sees a wall of data. There should be a brief guided tour that explains what each section means and why it matters.
  • Error recovery. If the API quota runs out or a channel is private, the error message is functional but not helpful. I would add suggestions like "Try again in 24 hours" or "Try this channel instead."
  • Share and collaborate. The share link feature generates a URL but it does not actually resolve to anything yet. This needs a backend to store and serve shared reports.
  • Keyboard shortcuts. Power users (the VP of Content Strategy who uses this daily) would benefit from keyboard navigation: J/K to move between videos, E to export, S to save.

Version 2 Vision

If this product gets traction after the Monday demo, version 2 should focus on three things:

  1. Persistence and accounts. Supabase or Postgres for storing analyses, user accounts, team workspaces. This unlocks saved history, trend tracking, and team collaboration.
  2. AI-generated recommendations. Feed the analysis data into an LLM to generate specific, actionable recommendations: "Based on MrBeast's format graveyard, there is an opportunity in challenge-format videos in the cooking niche. His last challenge video averaged 4.2x his channel average but he has not posted one in 8 months."
  3. Embeddable widgets. Let agencies embed live analysis widgets in their own dashboards or client presentations. Iframe-able charts, exportable data feeds, API access for custom integrations.

The core bet: Most YouTube analytics tools show you what happened. Vexel should tell you what to do about it. The Intelligence Brief and Format Graveyard are early steps in that direction. Version 2 doubles down on prescriptive intelligence over descriptive data.

05How I Used AI to Move Fast

AI was the accelerator. I was the driver.

My Workflow

I used Claude Code (Anthropic's CLI tool) as my primary development partner. The workflow was conversational and iterative. I would describe what I wanted in plain English, review what it generated, give feedback, and iterate. Some components landed on the first try. Others took 3-4 rounds of "not quite, try this instead."

The key thing: I never blindly accepted output. Every algorithm, every design choice, every architectural decision went through my judgment. When the outlier score was broken (every video showing 10/10), I identified the root cause (Shorts skewing the baseline) and directed the fix (separate baselines + log-scale). The AI wrote the code. I diagnosed the problem.

Where AI Saved the Most Time

  • Boilerplate and scaffolding. TypeScript interfaces, component prop types, API route handlers, utility functions. All of this is tedious but well-defined. AI generates it in seconds instead of me typing it out over an hour.
  • CSS implementation. I described the look I wanted ("frosted glass card with subtle border, 12px radius, hover lifts 4px"). The AI translated that into exact Tailwind classes and inline styles. Faster than looking up utility class names.
  • Regex patterns. The 24 curiosity gap patterns would have taken an hour to research and write. AI generated a solid first draft in seconds, and I added a few more based on patterns I have seen in real YouTube titles.
  • Three.js setup. The 3D particle background involved a lot of WebGL boilerplate (BufferGeometry, ShaderMaterial, Points, Lines). AI handled the setup, and I directed the aesthetics through multiple iterations.

Where I Overrode the AI

  • Information hierarchy. The AI initially put the video grid first on the analysis page. I moved Intelligence Brief to the top because that is what the buyer cares about. Data comes second to insight.
  • Scoring methodology. The original outlier score was a raw ratio. I pushed for log-scale normalization because I understood the data distribution issue. The AI implemented my approach, but the decision was mine.
  • Visual direction. We went through 4 different 3D backgrounds (crystalline gem, morphing sphere, orbital rings, particle network) before I settled on the one that felt right. The AI suggested options. I picked based on gut and product sense.
  • Export structure. The AI's first CSV was just a flat video list. I restructured it into 6 sections with channel summary, intelligence brief, win formula, format graveyard, detailed video data, and cadence heatmap. Because a CSV that a data analyst downloads should tell a complete story, not just be a data dump.

06Key Tradeoffs

What I prioritized, what I deferred, and why.

Ship Speed vs. Perfect Code

Some components have inline styles alongside Tailwind classes. In a longer project, I would enforce strict design token usage everywhere. For a Monday demo, visual consistency matters more than code purity. The product looks cohesive even if the implementation is not perfectly uniform under the hood.

Demo Polish vs. Production Robustness

Error handling is functional but not bulletproof. Edge cases like private channels, deleted videos, or malformed URLs are handled, but the error messages could be more helpful. Rate limiting, input sanitization, and abuse prevention are not implemented. For a client demo, these do not matter. For launch, they are essential.

Feature Breadth vs. Depth

I built 12 analysis features at moderate depth rather than 3 features at extreme depth. This was intentional. The Monday demo needs to impress across the board: "look at everything this does." A deep dive into one feature does not sell a SaaS product. Breadth sells the vision, and depth comes in version 2 once we know which features the client actually uses.

Mock Data vs. API-First

I built all UI against mock data first, then wired the API last. This meant I could iterate on design and layout without worrying about API failures, quota limits, or network latency. Every component worked perfectly before a single real API call was made. This is the fastest way to build a product like this, and it meant the API integration was essentially plug-and-play.

PDF Export via Print vs. Library

The PDF export uses the browser's native print functionality with custom print styles, not a library like jsPDF or Puppeteer. This was a conscious choice: print-to-PDF produces better-looking output than most JS PDF libraries, works offline, and does not add a 500KB dependency. The tradeoff is less control over page breaks and headers/footers. For the MVP, the result looks professional enough.

07Project Structure

Clean, predictable file organization following Next.js 14 conventions:

DirectoryPurpose
app/Next.js App Router pages and API routes
app/api/Server-side API routes (channel lookup, video fetch)
app/analysis/Results page with AnalysisClient component
app/export/Export page with PDF/CSV/JSON generation
src/components/18 reusable components, all named exports
src/lib/Core logic: analysis engine, YouTube API, utilities, mock data
src/types/TypeScript interfaces (Video, Channel, AnalysisResult, etc.)
public/Static assets (favicon)

Key architecture rule: src/lib/analysis.ts contains all scoring logic and makes zero network calls. src/lib/youtube.ts handles all API communication. They never import each other. This separation means the scoring engine is testable, portable, and free.
